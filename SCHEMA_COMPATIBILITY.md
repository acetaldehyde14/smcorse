## 3. Column conflicts between legacy files and canonical chain

### `telemetry_frames` — type differences

`setup-database.sql` and `src/db/migrate-telemetry.sql` defined many
`telemetry_frames` columns as `NUMERIC(n,m)`. Migration 004 redefines them
as `DOUBLE PRECISION`. Because migration 004 uses `CREATE TABLE IF NOT EXISTS`
followed by `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`, it will not change
the types of columns that already exist from `setup-database.sql`. This is
safe in practice (PostgreSQL can implicitly cast between the two in queries)
but means older installs may have `NUMERIC` columns where newer installs have
`DOUBLE PRECISION`. If strict type matching matters, run:

```sql
-- Check actual column types
SELECT column_name, data_type, numeric_precision, numeric_scale
FROM information_schema.columns
WHERE table_name = 'telemetry_frames' AND table_schema = 'public'
ORDER BY ordinal_position;
```

### `laps.lap_time` — type difference

`iracing-coach/database/schema.sql` defines `lap_time FLOAT`. Migration 002
adds `lap_time NUMERIC(10,3)` via `ALTER TABLE laps ADD COLUMN IF NOT EXISTS`.
On a database created from `schema.sql` first (the required flow), the column
already exists as `FLOAT` so the migration's `ADD COLUMN IF NOT EXISTS` is a
no-op. Runtime code reads the value correctly either way.

### `users.created_at` — `iracing-enduro-server/db/migrate.sql` only

`iracing-enduro-server/db/migrate.sql` adds `created_at TIMESTAMPTZ` to
`users`. Migration 001 does not add this column (it is assumed to exist from
`schema.sql`). `schema.sql` defines `created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP`
on `users`, so this is covered. No conflict.

### `stint_roster.actual_start_session_time` — `FLOAT` vs `DOUBLE PRECISION`

Legacy files use `FLOAT`. Migration 002 uses `DOUBLE PRECISION`. These are
aliases in PostgreSQL; no conflict.

## 5. Risks

### ~~RISK-1~~ (Resolved): `telemetry_sessions`, `telemetry_laps`, `telemetry_batches`

**Resolved by `migrations/010_live_session_pipeline.sql`.**

These tables are now part of the canonical chain. `npm run db:migrate` applies
migration 010 automatically. The manual `scripts/migrate.sql` workaround is no
longer required for new installs. Existing installs that already had these tables
from the old manual migration are safe — `CREATE TABLE IF NOT EXISTS` guards
ensure no double-create errors.

### RISK-2 (Medium): `reference_laps` and `coaching_sessions` not in canonical chain

Used by `src/routes/analysis.js` (coaching history endpoints) and
`src/routes/library.js` (reference lap browsing). These tables are defined in
`iracing-coach/database/schema.sql`, which is documented as a prerequisite for
migrations. As long as `schema.sql` is applied first, these tables exist. The
risk is that documentation may not always make this dependency explicit.

**Current mitigation**: README and `db/README.md` document `schema.sql` as
step 2 before `npm run db:migrate`.

### RISK-3 (Low): `user_preferences` and `user_progress` not in canonical chain

Defined in `iracing-coach/database/schema.sql`. No current route files query
these tables. Risk is that if routes are added that reference them, they must
still rely on `schema.sql` being applied first.

### RISK-4 (Low): `telemetry_frames` column type drift on old installs

See "Column conflicts" section above. Functionally safe; only a concern if
exact NUMERIC precision is required for an analytics query.
